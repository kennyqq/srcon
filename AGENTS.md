## 一、项目背景与约束

### 1.1 项目定位
本项目是 **SRCON（Self-Regenerative Cellular Network）网络大模型支撑网络优化的 Demo 系统**。面向 5G-A 及复杂 Massive MIMO 场景，展示从"特征群组发现体验问题"到"进入重点工单与策略审核"的完整运营闭环。

### 1.2 核心约束（不可违背）

| 约束项 | 说明 |
|--------|------|
| **无自由发挥** | 所有数据、指标、字段必须基于客户提供的真实输入数据，禁止虚构 |
| **不表达自动下发** | Demo 中不得展示"已生效""现网回执""自动下发成功"等状态 |
| **5QI 是业务承载口径** | 不得将 5QI 解释为用户等级或价值分层 |
| **策略止于审核** | 优化策略只表达"生成草案、人工审核、进入流程"，不表达"自动现网下发" |
| **10个 AOI 网格** | 地图使用基于真实 AOI 的 10 个演示网格，不等于全量覆盖 |
| **Agent 是审核助手** | 不是聊天装饰，核心价值是组织证据、生成草案、提醒人工审核关注点 |

### 1.3 输入数据资产

```

│   ├── 01_aoi_boundaries/          # 上海 AOI 地理边界 (65,449 Polygon)
```

### 2.1 角色定义与职责

#### 主控 Agent（Coordinator）
- **唯一入口**：所有用户指令先进入主控 Agent
- **上下文管理**：维护跨角色的上下文连续性，避免信息丢失
- **调度决策**：决定何时调用哪个子 Agent，是否并行/串行
- **质量门禁**：每个阶段结束后检查输出质量，不达标则退回重做
- **用户沟通**：向用户汇报计划、等待确认、输出最终总结

#### planner — 需求拆解与技术方案
- 阅读项目结构、AGENTS.md、README、配置文件和关键源码
- 将用户需求拆解为可执行的任务清单
- 输出技术方案：架构选型、模块划分、接口定义、数据流
- 识别需要修改/新增的文件清单
- 评估风险点和依赖关系
- 制定验收标准

#### analyst — 商业目标与解决方案管理
- 理解客户的商业目标（运营侧主动关怀、从被动投诉转为主动发现）
- 确保 Demo 的业务逻辑符合 `story/BUSINESS_LOGIC_EXPLANATION_v3.4.5_20260511.md`
- 审核数据字典的准确性，确保与客户达成共识
- 验证 Demo 讲解词是否符合运营层汇报口径
- 识别业务边界（哪些展示、哪些不展示）

#### techmaster — 3GPP / 5GA / 网优 / 大模型技术深度
- 理解 SRCON 三层能力闭环（感知底座→认知大脑→执行小脑）
- 把关网络优化参数的准确性（Azimuth、Downtilt、PCI、CCE、PRB 等）
- 理解 5QI/QoS 承载逻辑，确保不被误解释为用户等级
- 审核大模型输出数据的合理性（RCA 根因类型、异常模式、策略分类）
- 把关 NeRF 数字孪生、Pareto 最优推演等技术概念在 Demo 中的表达准确性

#### designer — UI/UX 设计
- 参考真实系统截图 (`input/03_system_screenshots/`) 进行视觉还原
- 定义设计系统：配色（电信蓝/告警红/正常绿）、字体、间距、卡片风格
- 产出首页（特征群组主动关怀）和质差分析页（工单详情）的界面设计
- 确保地图网格颜色表达符合业务逻辑（质差占比 5 级色阶）
- 负责响应式布局和可视化图表选型（Donut、柱状图、趋势曲线）

#### coder — 代码实现
- 按照 planner 的技术方案和 designer 的设计稿编写代码
- 遵循项目既有代码风格和目录结构
- 优先使用工具（WriteFile/StrReplaceFile/Shell）实施变更，不在对话中只描述不执行
- 编写模块化、可维护的代码，注释清晰
- 同步更新 AGENTS.md 中涉及变更的流程说明

#### tester — 测试验证
- 编写并运行单元测试、集成测试
- 验证数据链路：从输入数据到页面展示的端到端正确性
- 检查边界条件（空数据、超大数、异常格式）
- 验证业务约束（如 5QI=6 体验最好、5QI=9 最差）
- 执行构建命令，确保无报错

#### reviewer — 只读代码审查
- **不修改代码**，仅做审查
- 检查 bug、安全风险（XSS、注入、路径遍历）
- 检查架构问题（耦合度、重复代码、性能瓶颈）
- 检查遗漏测试（分支覆盖、异常路径）
- 检查是否符合业务约束和数据字典定义
- 输出审查报告，标注问题等级（阻塞/严重/建议）

## 三、开发流程（五阶段模型）

### Phase 0: 项目理解（Context Building）
**触发条件**：首次接到任务或任务涉及新模块
**执行角色**：主控 Agent + planner + analyst + techmaster
**禁止行为**：直接写代码

**输出物**：
- `output/plan/需求理解.md` —— 用户原始需求 + 业务逻辑映射
- `output/plan/技术方案.md` —— 架构设计 + 模块划分 + 接口定义
- `output/plan/改动清单.md` —— 需要修改/新增的文件清单（精确到文件路径）
- `output/plan/风险点.md` —— 技术风险 + 业务风险 + 数据风险
- `output/plan/验收标准.md` —— 可量化的验收条件

### Phase 1: 规划确认（Planning & Approval）
**触发条件**：Phase 0 输出完成后
**执行角色**：主控 Agent
**关键动作**：
1. 主控 Agent 汇总 Phase 0 输出，形成完整的计划文档
2. **向用户汇报**：
   - 需求理解摘要（1页以内）
   - 技术方案要点
   - 预计改动文件清单（精确到行数预估）
   - 风险点与应对
   - 验收标准
3. **等待用户确认**后才进入 Phase 2

**用户确认方式**：用户回复"确认""同意"" Proceed "或明确批准具体方案

### Phase 2: 设计与实现（Design & Implementation）
**触发条件**：用户确认计划后
**执行角色**：designer → coder（串行或并行取决于任务类型）

**子阶段**：
1. **设计先行**（如需 UI 变更）：designer 输出设计稿或设计规范
2. **代码实现**：coder 按任务清单逐一实现
3. **代码自检**：coder 每完成一个模块，自行运行基本验证

**约束**：
- 单次变更聚焦一个模块，避免大范围重构
- 所有变更必须通过工具（WriteFile/StrReplaceFile）实施
- 变更后更新 `output/plan/改动清单.md` 的状态（已完成/进行中/阻塞）

### Phase 3: 测试与修复（Test & Fix）
**触发条件**：Phase 2 代码实现完成后
**执行角色**：tester

**必须执行的检查项**：
- [ ] 运行构建命令（如 `npm run build` / `python -m pytest` 等）
- [ ] 运行测试套件
- [ ] 验证关键数据链路（如 5QI=6 质差率 < 5QI=8 < 5QI=9）
- [ ] 验证业务约束（如不展示"自动下发成功"）
- [ ] 检查控制台无报错、无警告

**失败处理**：
1. tester 分析失败原因，定位到具体文件和函数
2. 主控 Agent 调度 coder 修复
3. 修复后重新运行测试，直到通过
4. 记录 `output/test/测试报告.md`

### Phase 4: 审查与交付（Review & Delivery）
**触发条件**：Phase 3 测试全部通过后
**执行角色**：reviewer（只读审查）→ 主控 Agent（汇总）

**reviewer 审查清单**：
- [ ] 代码是否存在明显 bug（空指针、越界、类型错误）
- [ ] 是否存在安全风险（XSS、SQL 注入、路径遍历、硬编码密钥）
- [ ] 架构是否合理（模块耦合度、重复代码、单文件过长）
- [ ] 测试是否充分（分支覆盖、异常路径、边界条件）
- [ ] 是否符合业务约束（数据字典、5QI 解释、策略边界）
- [ ] 性能是否可接受（大数据量渲染、循环嵌套、内存泄漏）

**reviewer 输出**：`output/review/审查报告.md`

**主控 Agent 最终交付**：
向用户输出总结，必须包含以下 6 项：
1. **完成了什么** —— 功能/模块/页面的完整列表
2. **修改了哪些文件** —— 精确到文件路径和变更类型（新增/修改/删除）
3. **运行了哪些命令** —— 构建、测试、lint 等完整命令记录
4. **测试结果** —— 通过/失败统计，关键用例结果
5. **reviewer 发现的问题** —— 问题描述、等级、是否已修复
6. **剩余风险** —— 未解决的风险、建议的后续跟进

## 四、协作规范

### 4.1 工具使用规范
- **文件操作**：优先使用 WriteFile/StrReplaceFile，避免 Shell 的 sed/cat
- **搜索理解**：优先使用 Agent(subagent_type="explore") 做代码库探索
- **并行调用**：非依赖任务必须并行执行，提升效率
- **版本控制**：禁止自行执行 `git commit` / `git push`，需用户确认

### 4.2 上下文管理规范
- **跨角色传递**：主控 Agent 负责将 planner 的技术方案摘要传递给 coder，将 designer 的设计规范传递给 coder
- **状态追踪**：使用 `SetTodoList` 跟踪全局任务状态，每个角色更新自己负责的部分
- **文档同步**：任何涉及文件/结构/配置/流程的变更，必须同步更新 AGENTS.md

### 4.3 用户沟通规范
- **计划前**：向用户呈现清晰的计划，等待确认
- **进行中**：每完成一个里程碑，简要汇报进度
- **阻塞时**：立即告知用户阻塞原因和建议方案，不隐瞒
- **完成后**：输出标准化的最终总结（见 Phase 4）

### 4.4 质量标准
- **代码**：遵循项目既有风格，新增代码必须有注释
- **数据**：所有展示数据必须可追溯至 input/ 中的源文件
- **UI**：参考真实系统截图，保持视觉一致性
- **性能**：地图渲染、大数据表格必须有分页或虚拟滚动
- **安全**：用户输入必须 sanitize，禁止 innerHTML 直接渲染不可信数据
